下面是从这份 ModelGate 评估里整理出的 **所有发现问题** ，我按优先级分成 P0 / P1 / P2，方便你后面直接改。来源是你上传的评估文本。

## P0：最优先解决的问题

| 问题                                                  | 影响                                                                                            | 建议                                                                                              |
| ----------------------------------------------------- | ----------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| **API Key 明文存储在 `provider_secrets`表里** | 这是最大的安全风险。即使是本地单用户项目，只要数据库或磁盘镜像泄露，所有厂商 API Key 都会泄露。 | 改成 OS keystore / keyring 派生密钥 + AES-GCM 加密存储。                                          |
| **没有 CI / GitHub Actions**                    | lint、typecheck、test、E2E 全靠人工跑，后期很容易质量回退。                                     | 加 `.github/workflows/ci.yml`，串起 `ruff`、`pytest`、`tsc`、`eslint`、`playwright`。 |
| **流式输出未做**                                | 对聊天类产品用户感知非常强，没有 streaming 会显得“不像成熟 AI 工具”。                         | 优先做 Chat Runtime 的 streaming response。                                                       |
| **运行中取消未做**                              | 用户发错请求、生成太慢、视频任务卡住时无法中断，体验差。                                        | 给 Chat / Generation 都加 cancel 机制。                                                           |
| **多模态图片未接入**                            | 项目定位是多模型工作台，但图片识别没接上会削弱核心价值。                                        | 先接 chat + image input，再扩展复杂多模态。                                                       |
| **Seedance 真实接入和二进制结果持久化未做**     | 视频/图片生成是用户很关心的功能，如果只是架构预留，第一版体验会偏弱。                           | 优先完成 Seedance smoke test、任务状态轮询、结果文件保存。                                        |

## P1：重要但可以排在 P0 后面

| 问题                                                | 影响                                                         | 建议                                                                                               |
| --------------------------------------------------- | ------------------------------------------------------------ | -------------------------------------------------------------------------------------------------- |
| **README 的核心价值主张不够锐利**             | 用户可能看不出 ModelGate 和“自己写个 API 小客户端”的区别。 | README 开头直接强调：多 Provider、多模型能力路由、动态参数面板、历史记录、异步生成、统一 Adapter。 |
| **Celery + Redis 对本地单用户略重**           | 增加部署复杂度，可能让个人用户启动成本变高。                 | 保留 Celery 也可以，但文档里解释为什么需要；或者提供轻量 dev mode。                                |
| **缺少对象存储抽象层**                        | 未来从本地 storage 切到 S3 / R2 / OSS 时，可能要改业务代码。 | 抽一层 `StorageAdapter`，本地文件只是其中一种实现。                                              |
| **缺少顶层“架构图 + 数据流图”**             | 现在文档很多，但新人要自己拼完整心智模型。                   | 新增一份 `Architecture Overview.md`，包含请求流、模型路由流、异步生成流、文件上传流。            |
| **没有 mypy / black / prettier / pre-commit** | 现在代码质量不错，但缺少长期约束机制。                       | 后端加 `mypy + black`，前端加 `prettier`，根目录加 `pre-commit`。                            |
| **前端没有 stylelint**                        | CSS / 样式规范缺少自动检查。                                 | 加 stylelint 或至少用 prettier 管理样式格式。                                                      |
| **后端没有 CI**                               | 后端测试和 lint 不能自动拦截问题。                           | 并入 GitHub Actions。                                                                              |

## P2：后续完善项

| 问题                                                   | 影响                                                                         | 建议                                                                     |
| ------------------------------------------------------ | ---------------------------------------------------------------------------- | ------------------------------------------------------------------------ |
| **下载 / 预览鉴权推迟到多用户版本**              | 本地单用户问题不大，但如果之后开放网络访问，会变成安全坑。                   | 提前设计不可猜测 ID、文件权限校验、临时下载链接。                        |
| **不可猜测 ID 和文件权限校验未优先完成**         | 文件访问路径容易留下隐患。                                                   | 至少先把 public path 和真实存储 path 隔离。                              |
| **API Key 安全文档偏轻**                         | 其他模块文档很完整，但密钥安全文档只有较少内容，和项目整体严谨度不匹配。     | 单独补充 threat model、存储方案、密钥轮换、日志脱敏、备份风险。          |
| **没有测试覆盖率报告**                           | 不知道现有测试到底覆盖了多少关键路径。                                       | 后端加 `coverage.py`，前端加 Istanbul / Vitest coverage。              |
| **没有契约测试**                                 | Provider Adapter、前后端 API、模型配置 JSON 和 DB schema 之间容易不一致。    | 给 API schema、Provider response、Model Registry 配置加 contract tests。 |
| **Generation Runtime 状态机并发 / 竞态测试不足** | 异步任务最容易出问题，比如重复提交、取消后完成、超时和回调乱序。             | 专门补并发测试、重复任务测试、取消任务测试、过期任务测试。               |
| **没有结构化 metrics**                           | 后期不好观察调用耗时、失败率、token 消耗、任务成功率。                       | 加 Prometheus 或 OpenTelemetry。                                         |
| **没有日志聚合方案**                             | 本地还好，部署后排错会困难。                                                 | 先统一 JSON log，再考虑 Loki / OpenTelemetry collector。                 |
| **`.env.example`和代码配置没有接入一致性校验** | 配置字段容易遗漏，别人部署容易踩坑。                                         | 把已有配置校验脚本接入 CI。                                              |
| **Phase 10 高级能力未启动**                      | Workflow、多模型对比、OpenAI-compatible endpoint、MCP Server、多用户还没做。 | 这不算当前版本硬伤，但建议先补完 P0 用户感知功能再进入 Phase 10。        |

## 最核心的修改顺序

我建议你按这个顺序改：

1. **API Key 加密存储**
2. **GitHub Actions CI**
3. **流式输出**
4. **运行中取消**
5. **图片多模态接入**
6. **Seedance 真实接入 + 结果持久化**
7. **架构图 / 数据流图**
8. **测试覆盖率 + 状态机并发测试**
9. **对象存储抽象**
10. **README 核心价值重写**

一句话总结：
目前 ModelGate 最大的问题不是架构不行，而是**安全底座、自动化质量保障、用户强感知功能**这三块还没补齐。先把这三类补上，项目观感会从“架构很完整的个人项目”提升到“可以认真使用的产品级工具”。
