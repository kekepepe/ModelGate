可以。我们先把它定义成一个完整产品：

# 多模型任务工作台：完整设计方案 + 技术框架

你的目标不是做一个普通聊天网页，而是做一个：

> **任务优先 + 模型能力驱动 + 多 Provider API 管理 + 多模态工作台**

核心原则是：

```text
用户先选择任务，而不是先选择模型。
系统根据任务、上传文件类型、模型能力，自动过滤可用模型。
不同类型模型走不同调用链：
Chat / Coding / Vision 一条线；
Image / Video 生成另一条线。
```

---

# 1. 产品定位

## 产品名称暂定

```text
AI Model Workspace
多模型任务工作台
```

## 解决的问题

你现在的问题是：

```text
1. 有多个 API plan：MiniMax、火山、MiMo、Kimi、GLM 等
2. 每个 plan 下面有很多模型
3. 官方模型名难记、难查
4. 不同模型能力不同：文本、代码、识图、生图、生视频、图生视频
5. 不同任务需要不同参数面板
6. 多 Agent 使用时，不想反复切 API、模型、工具
```

所以这个 app 的核心价值是：

```text
1. 统一管理所有 provider 和模型
2. 直接显示官方模型原名
3. 根据任务自动推荐模型
4. 根据上传文件类型自动过滤模型
5. 根据模型能力动态生成操作界面
6. 区分 Chat/Coding/Vision 与 Image/Video 两类模型
7. 后续可以扩展多 Agent 工作流
```

# 3. 核心设计理念

## 3.1 任务优先

首页不应该先让你选模型，而是先让你选任务。

```text
任务中心
├── 聊天 / 文本生成
├── 写代码
├── 代码审查
├── 图片理解
├── 文档分析
├── 视频理解
├── 文生图
├── 图生图
├── 文生视频
├── 图生视频
├── 首尾帧视频
├── 分镜生成
├── 广告脚本
├── Prompt 优化
└── 多 Agent 工作流
```

选完任务后，系统再根据这个任务展示可用模型。

例如你选择：

```text
任务：图生视频
```

系统自动只显示：

```text
支持 image_to_video 的模型
```

你选择：

```text
任务：图片理解
```

系统自动只显示：

```text
支持 vision_understanding 的模型
```

---

## 3.2 保留官方模型原名

你现在不需要模型别名，所以模型选择区应该直接显示官方模型名。

例如：

```text
Provider: 火山引擎
Model: seedance-xxx-xxx
```

```text
Provider: Moonshot / Kimi
Model: kimi-k2.6-xxx
```

```text
Provider: Zhipu / GLM
Model: glm-5.1-xxx
```

但是可以在旁边展示解释信息：

```text
官方模型名：seedance-xxx-xxx
Provider：火山引擎
能力：文生视频、图生视频
输入：文本、图片
输出：视频
```

也就是说：

```text
模型原名必须显示
但可以加辅助标签帮助选择
```

---

## 3.3 根据上传文件类型过滤模型

这是非常重要的体验优化。

上传文件后，模型列表自动变少。

### 示例 1：没有上传文件

可选任务：

```text
聊天
写代码
文生图
文生视频
Prompt 优化
```

可选模型：

```text
文本模型
代码模型
图像生成模型
视频生成模型
```

---

### 示例 2：上传图片

系统识别文件类型：

```text
input_type = image
```

然后自动过滤模型：

```text
保留支持以下能力的模型：
- vision_understanding
- image_to_image
- image_to_video
- image_caption
```

界面显示：

```text
可用任务：
- 图片理解
- 图生图
- 图生视频
- 图片转 Prompt
```

不显示纯文本模型，除非允许“图片转文字后再用文本模型处理”。

---

### 示例 3：上传视频

系统识别：

```text
input_type = video
```

可用任务变成：

```text
- 视频理解
- 视频总结
- 视频转分镜
- 视频分析
```

模型列表只保留：

```text
支持 video_understanding 的模型
```

---

### 示例 4：上传 PDF / Word

系统识别：

```text
input_type = document
```

可用任务：

```text
- 文档总结
- 文档问答
- 提取大纲
- 改写
- 翻译
- 生成 PPT 大纲
```

模型列表保留：

```text
支持 file_input 或 long_context 的模型
```

如果模型不支持原生文件输入，则系统可以走：

```text
文件解析 → 文本提取 → 发送给文本模型
```

# 4. 模型分类设计

你的系统里模型应该先分成两大类。

## 4.1 Chat / Coding / Vision 模型

这类模型本质上是“输入内容，输出文本”。

包括：

```text
文本聊天模型
代码模型
视觉理解模型
文档理解模型
视频理解模型
长上下文模型
Agent 模型
```

统一走：

```text
Chat Runtime
```

典型接口形式：

```text
POST /api/chat/completions
```

输入：

```json
{
  "model": "官方模型名",
  "provider": "provider_id",
  "messages": [],
  "files": [],
  "temperature": 0.7,
  "max_tokens": 4096
}
```

输出：

```json
{
  "type": "text",
  "content": "模型回复内容"
}
```

---

## 4.2 Image / Video 生成模型

这类模型本质上是“提交任务，生成媒体文件”。

包括：

```text
文生图
图生图
文生视频
图生视频
首尾帧视频
视频增强
视频扩展
```

统一走：

```text
Generation Runtime
```

这类模型通常不是一次请求马上返回结果，而是：

```text
创建任务 → 轮询状态 → 获取结果 → 展示 / 下载
```

典型接口：

```text
POST /api/generation/tasks
GET  /api/generation/tasks/:taskId
GET  /api/generation/tasks/:taskId/result
```

输入：

```json
{
  "provider": "volcengine",
  "model": "seedance-官方模型名",
  "taskType": "text_to_video",
  "prompt": "山村小学教室里...",
  "params": {
    "duration": 5,
    "aspect_ratio": "16:9",
    "resolution": "1080p"
  }
}
```

输出：

```json
{
  "taskId": "local_task_id",
  "providerTaskId": "provider_task_id",
  "status": "processing"
}
```

# 7. Capability Router 设计

Capability Router 是自动推荐模型的核心。

它的输入：

```json
{
  "taskType": "image_to_video",
  "uploadedFiles": [
    {
      "type": "image",
      "mime": "image/png"
    }
  ],
  "preferredProviders": ["volcengine"],
  "requiredOutput": "video"
}
```

它的输出：

```json
{
  "availableModels": [
    {
      "officialModelName": "seedance-xxx-xxx",
      "provider": "volcengine",
      "reason": "支持 image_to_video，输入 image，输出 video"
    }
  ],
  "hiddenModels": [
    {
      "officialModelName": "kimi-k2.6-xxx",
      "reason": "该模型支持图片理解，但不支持视频生成"
    }
  ]
}
```

---

## 7.1 过滤规则

```text
第一步：根据 taskType 过滤
第二步：根据 inputTypes 过滤
第三步：根据 outputTypes 过滤
第四步：根据 enabled 状态过滤
第五步：根据 provider 可用状态过滤
第六步：根据参数支持情况过滤
```

伪代码：

```ts
function recommendModels(taskType, files, registry) {
  const inputTypes = detectInputTypes(files)

  return registry.models.filter(model => {
    if (!model.enabled) return false
    if (!model.taskTypes.includes(taskType)) return false

    for (const input of inputTypes) {
      if (!model.inputTypes.includes(input)) return false
    }

    return true
  })
}
```

# 10. Provider Adapter 设计

每个供应商一个 adapter。

```text
adapters/
├── minimax.adapter.ts
├── volcengine.adapter.ts
├── mimo.adapter.ts
├── moonshot.adapter.ts
├── zhipu.adapter.ts
└── base.adapter.ts
```

每个 adapter 实现统一接口。

```ts
interface ProviderAdapter {
  chat(input: ChatInput): Promise<ChatOutput>
  createGenerationTask?(input: GenerationInput): Promise<TaskOutput>
  getGenerationTask?(taskId: string): Promise<TaskStatus>
  cancelGenerationTask?(taskId: string): Promise<void>
  listModels?(): Promise<ModelInfo[]>
}
```

这样上层不关心具体供应商 API 怎么写。

例如火山视频模型：

```ts
volcengineAdapter.createGenerationTask()
volcengineAdapter.getGenerationTask()
```

Kimi / GLM / MiMo 文本视觉模型：

```ts
moonshotAdapter.chat()
zhipuAdapter.chat()
mimoAdapter.chat()
```

# 17. 多 Agent 工作流预留设计

虽然第一阶段可以不做，但框架里可以预留。

多 Agent 本质是多个 task node 串联或并行。

例如你的视频广告流程：

```text
剧本 Agent → 分镜 Agent → Prompt Agent → 生图 Agent → 生视频 Agent → 审查 Agent
```

可以设计为：

```json
{
  "workflowId": "ad_video_workflow",
  "nodes": [
    {
      "id": "script_agent",
      "taskType": "storyboard",
      "model": "kimi-k2.6-xxx"
    },
    {
      "id": "prompt_agent",
      "taskType": "prompt_optimize",
      "model": "minimax-xxx"
    },
    {
      "id": "video_agent",
      "taskType": "text_to_video",
      "model": "seedance-xxx"
    }
  ],
  "edges": [
    ["script_agent", "prompt_agent"],
    ["prompt_agent", "video_agent"]
  ]
}
```

后面可以加一个：

```text
Workflow Builder
```

界面像这样：

```text
[输入需求]
   ↓
[脚本生成]
   ↓
[分镜生成]
   ↓
[视频 Prompt 优化]
   ↓
[Seedance 生成视频]
   ↓
[结果审查]
```


# 23. 最关键的设计取舍

你现在这个产品一定要坚持这几个原则：

```text
1. 任务优先，不是模型优先
2. 显示官方模型原名，不强制别名
3. 模型能力配置化，不写死
4. 参数面板动态生成，不为每个模型单独写页面
5. Chat/Coding/Vision 和 Image/Video 分开运行
6. Seedance 这类视频模型走异步任务系统
7. API Key 只放后端
8. 每个供应商一个 adapter
9. 所有历史记录都保存：prompt、模型、参数、结果
10. 后续预留多 Agent Workflow
```
