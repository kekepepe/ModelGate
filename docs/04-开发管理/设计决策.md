### D-2026-06-06-02 — 测试 fixture 必须 monkeypatch `startup_module.engine`/`SessionLocal`

**决策：** 任何用 `TestClient(app)` + SQLite StaticPool 的 self-contained fixture，必须同时 `monkeypatch.setattr` 三处：`session_module.{engine, SessionLocal}` + `startup_module.{engine, SessionLocal}` + 用到 module-level `SessionLocal` 的服务模块（如 `provider_secrets_module.SessionLocal`）。

**Why:**
- `app/core/startup.py` 第 9 行 `from app.db.session import SessionLocal, engine` 在 import 时把对象绑死到局部名，单独 patch `session_module.engine` 不会改变 startup 的 `engine` 引用，lifespan 仍会去连真实 Postgres。
- `provider_secrets.py` 同样在 module 顶部 `from app.db.session import SessionLocal`，`get_local_provider_secret(db=None)` 走的是这条 import 时绑定的 SessionLocal。
- phase10 的 fixture 之所以"看似工作"是因为 5432 端口在某些环境恰好被本地 Postgres 占着；换到没 Postgres 的机器就会全 ERROR。phase11 第一次跑确实就是这个错。

**How to apply:** 新增使用 `TestClient + 内存 SQLite` 的 phase 测试，把这条 monkeypatch 三件套抄上；如果服务模块顶部还有 import-time 绑定 SessionLocal 的，再加一行。

## 当前任务 D-2026-06-07-01 — `/activity` 加 `<Suspense>` 包裹（修 Docker build 报错）

### D-2026-06-07-01 — `/activity` 用 `<Suspense fallback={null}>` 包裹，与 `/api-keys` / `/workspace` 同构

**决策：** `apps/web/src/app/activity/page.tsx` 把 `<ActivityPage />` 用 `<Suspense fallback={null}>` 包裹，import 从 `react` 引入 `Suspense`。`ActivityPage` 内部不改。

**Why:**
- 报错根因：`ActivityPage` 在顶层调用 `useSearchParams()`（读 `?runId=` 深链），Next.js 15 App Router 的静态预渲染要求所有 `useSearchParams` 调用点都必须在 `<Suspense>` 边界内；`/activity` 是 V2 P0 Stage 3 新增的，漏写了。
- 项目已有同构范式：`/workspace` 用 `<Suspense fallback={<main>...Loading Playground...</main>}>` 包 `WorkspaceShell`；`/api-keys` 用 `<Suspense fallback={null}>` 包 `ApiKeysClient`。`/activity` 走第 3 条同构路径。
- 选择 `fallback={null}` 而非带 skeleton：Activity 页面是表格 + 过滤器 + 详情抽屉 3 大块，首屏本来就要等 `useQuery` 拉取 `/history/runs` + `/logs/requests` + `/providers`，再加 skeleton 也只是"先空白再空白"，不如直接 `null`。
- **不**做"把 `useSearchParams` 拆到独立的子组件再用 Suspense 包那个子组件"那种最小化拆分：项目已有约定就是整页客户端组件外层包 Suspense，拆分只会破坏风格一致性、引入新的组件层级而无实际收益。
- **不**改 `next.config.ts` 加 `experimental.optimizePackageImports` 之类配置：报错是 build-time 静态预渲染规则，与打包优化无关。
- **不**给 `/activity` 加 `export const dynamic = "force-dynamic"`：会失去 SSG，与 `/workspace` / `/api-keys` 同样保持静态预渲染。

**How to apply:** 任何新 page 只要内部组件调用了 `useSearchParams` 或 `usePathname` 之外的动态导航 hooks，**必须**用 `<Suspense>` 包裹整个客户端组件作为唯一出口。Suspense fallback 选 `null` 还是带 skeleton，按"页面是否能利用 fallback 占位避免布局抖动"判断：表格/列表用 `null`，有明显 hero 区块的可以用 skeleton。

### D-2026-06-07-03 — Compare 走前端并行 `Promise.all`、不引入后端 batch endpoint

**决策：** Compare 多模型对比的"并行调用 N 个 run"完全在前端实现：`CompareDrawer` 调 N 次现有 `streamChatRun`，用 `Promise.all` 等所有完成；每个 slot 通过 `params._compare_group_id` 字段透传一个 UUID，后端写入 `runs.metadata.compare_group_id`（现有 jsonb 字段，零 schema 变更）。

**How to apply:** 类似"批量执行 + 收集结果"的前端场景（例如未来"批量测试 5 个 Provider"），优先复用单 endpoint × Promise.all，除非有"跨 run 的原子性 / 事务"硬需求。

### D-2026-06-07-05 — Preset 走 schema-aware 字段映射、不强制 Provider 一致

**决策：** 4 个 builtin Preset（Creative / Precise / Long context / Low cost）定义为"target 字段值"集合；apply 时按当前模型的 `paramsSchema` 做映射：字段不存在 → skip 不报错，非数字字段 → skip，超 min/max → clip 到边界。`long_context` 的 `max_tokens` 按 `schema.max * 0.8` 向下取整到 256 倍数。

**Why:**

**How to apply:** 未来加自定义 Preset（设计文档 §18.4，本次未实施）走完全相同的映射策略；新增 Provider adapter 时只要 schema 字段名对齐通用约定（`temperature`、`top_p`、`max_tokens`、`presence_penalty`、`frequency_penalty`），Preset 自动可用。

## 当前任务 D-2026-06-09-04..06 — 修复 Delete project run 在 Postgres 上点击无反应

### D-2026-06-09-04 — `delete_project_run` 改成按 FK 拓扑顺序删除，先 NULL 自引用

**决策：** `apps/server/app/api/projects.py::delete_project_run` 改为：先删 `artifacts`（引用 agent_runs/tasks）→ `agent_runs`（引用 tasks）→ `project_memory`，再把该 run 下所有 `project_tasks.parent_task_id` UPDATE 为 NULL，再 bulk delete `project_tasks`，最后 `db.delete(pr)`。所有 bulk 操作加 `synchronize_session=False`。

**Why:**
- 原实现 `db.query(ProjectTask).filter(...).delete()` 在 Postgres 上会触发 `ForeignKeyViolation`：`project_tasks.parent_task_id` 是自引用 FK（`models.py:318`），且 alembic 0004 创建 FK 时**没有 `ondelete="CASCADE"`**。bulk DELETE 不会按依赖顺序删除行，含 Planner 拆出的子任务（`parent_task_id` 非空）就会失败，返回 500，前端表现为"点击 Delete 无反应"。
- SQLite 默认关闭 FK，所以历史 phase17/phase18 测试没暴露这个 bug。
- 不动 schema（加 `ondelete="CASCADE"` 需要新 migration 重建外键，alter table 对已部署数据有风险），ORM 层 ordered delete 已经足够。

**How to apply:** 任何新增引用 `project_runs.id` 或 `project_tasks.id` 的表都要在 `delete_project_run` 里按依赖顺序提前删；任何在 `project_tasks` 上引入新自引用列都要在删除前 NULL 掉。

### D-2026-06-09-19 — ReactFlow pane 拦截点击的 E2E workaround

**决策：** Playwright E2E 测试中，ReactFlow 内节点按钮的点击使用 `page.evaluate(() => element.click())` 绕过 pane 拦截，且在点击前先 `await expect(board).toBeVisible()` 等待渲染完成。

**How to apply：** 新增含 ReactFlow 节点交互的 E2E 测试时，统一用 `page.evaluate` + `awaitVisible` 模式。

### D-2026-06-10-03 — V3.1 assistant message 走独立 `messages` slice，不复用 `latestRun`

### D-2026-06-10-04 — `messages.conversation_id` FK 无 `ondelete="CASCADE"`

**决策：** `messages` 表的 `conversation_id` FK 不加 `ondelete="CASCADE"`，删除 conversation 时在代码中先删 messages 再更新 conversation status。

**Why:**
- V2.5 Postgres delete bug 教训：`project_tasks` 的 `parent_task_id` self-referential FK + CASCADE 导致 Postgres 上删除失败。SQLite 默认忽略 FK 约束，测试环境无法复现。
- 当前项目所有 FK 均无 `ondelete="CASCADE"`（见 `0004_project_mode_tables.py`），删除顺序统一在 API 层代码控制。
- `messages` 表数据量可能很大，CASCADE DELETE 在高并发下有锁表风险。

**How to apply:**
- 后续若做 schema 大版本改造，可统一给所有 FK 加 CASCADE（见任务清单低优先级项）；当前保持一致性。

### D-2026-06-10-05 — conversation 软删（status="deleted"）+ messages 硬删

**决策：** `DELETE /conversations/{id}` 将 `conversations.status` 设为 `"deleted"`（软删），同时硬删关联的 messages。

**Why:**
- 软删 conversation 可保留审计 trail（created_at / updated_at / title），同时 list 接口自动过滤 `status="active"`。
- messages 硬删是因为消息内容本身无独立审计价值，且保留会占用大量存储。
- 与 Project Mode 的 delete 模式一致（`project_runs` 无软删但 `DELETE` 是 ordered bulk delete）。

**How to apply:**
- 后续若需"恢复已删除 conversation"功能，只需将 status 改回 "active"（但 messages 已硬删，需要额外设计）。

### D-2026-06-10-06 — stream endpoint 集成 conversation 持久化，不走独立 messages API

**决策：** `POST /chat/runs/stream` 接受可选 `conversationId`，在 stream 开始前创建 user message + assistant message（streaming），stream 结束后更新 assistant message。不新建独立的 `/conversations/{id}/messages/stream` endpoint。

**Why:**
- 减少 API 表面积：V3.1 的 `streamChatRun` 前端调用链已成熟，只需加一个 `conversationId` 参数即可复用。
- 独立 messages API 需要前端维护两套 stream 逻辑（有 conversationId / 无 conversationId），增加复杂度。
- SSE done event 中回传 `conversationId` + `messageId`，前端通过 URL 更新实现刷新恢复。

**How to apply:**
- 后续若需"在已有 conversation 中插入 system message"等高级场景，再拆分独立 messages API。
- `runs.metadata_json` 中的 `conversationId` / `messageId` 可用于 Details Drawer 反查。
