# 任务状态机设计文档

项目名称：ModelGate  
版本：v1.0  
覆盖范围：Chat Run、Generation Task、文件解析任务、Worker 轮询与取消语义

---

## 1. 设计目标

任务状态机用于保证任务状态可追踪、可恢复、可校验，避免前端、API、Worker 和 Provider 同时修改状态导致混乱。

原则：

- 状态最终写入 PostgreSQL。
- Redis 只用于队列、锁和短期进度。
- 前端不能直接修改任务状态。
- Worker 更新状态前必须校验当前状态。
- 所有状态流转必须有明确原因。
- Provider 原始状态必须映射成本地标准状态。

## 2. Chat Run 状态机

### 2.1 状态

```text
queued
running
completed
failed
cancelled
```

### 2.2 状态说明

| 状态 | 说明 |
|---|---|
| queued | run 已创建，等待执行 |
| running | 正在调用 Provider 或流式输出 |
| completed | 成功完成 |
| failed | 执行失败 |
| cancelled | 用户取消或系统取消 |

### 2.3 允许流转

```text
queued -> running
queued -> cancelled
running -> completed
running -> failed
running -> cancelled
```

禁止流转：

```text
completed -> running
completed -> failed
failed -> running
cancelled -> running
```

### 2.4 触发者

| 流转 | 触发者 |
|---|---|
| queued -> running | Chat Runtime |
| queued -> cancelled | API cancel |
| running -> completed | Chat Runtime |
| running -> failed | Chat Runtime / Provider Adapter |
| running -> cancelled | API cancel / AbortController |

## 3. Generation Task 状态机

### 3.1 状态

```text
queued
submitted
processing
completed
failed
cancelled
expired
```

### 3.2 状态说明

| 状态 | 说明 |
|---|---|
| queued | 本地任务已创建，尚未提交 Provider |
| submitted | 已提交 Provider，已获得或等待 providerTaskId |
| processing | Provider 正在处理 |
| completed | Provider 完成，输出已保存或可访问 |
| failed | Provider 或本地处理失败 |
| cancelled | 本地取消，Provider 可能已取消或仍在执行 |
| expired | Provider 或本地超时 |

### 3.3 允许流转

```text
queued -> submitted
queued -> cancelled
submitted -> processing
submitted -> failed
submitted -> cancelled
submitted -> expired
processing -> completed
processing -> failed
processing -> cancelled
processing -> expired
```

禁止流转：

```text
completed -> processing
completed -> failed
failed -> processing
expired -> processing
cancelled -> processing
```

## 4. Provider 状态映射

### 4.1 火山视频生成

Provider 状态：

```text
queued
running
succeeded
failed
expired
```

映射：

| 火山状态 | ModelGate 状态 |
|---|---|
| queued | submitted |
| running | processing |
| succeeded | completed |
| failed | failed |
| expired | expired |

### 4.2 通用异步 Provider

| Provider 状态类型 | ModelGate 状态 |
|---|---|
| pending / queued | submitted |
| running / processing | processing |
| success / succeeded / completed | completed |
| error / failed | failed |
| timeout / expired | expired |
| cancelled / canceled | cancelled |

Provider 原始状态必须保存到 `provider_status`。

## 5. 幂等提交

### 5.1 Idempotency-Key

所有写接口预留：

```http
Idempotency-Key: client_generated_key
```

后端行为：

- 如果 key 首次出现，正常创建任务。
- 如果 key 已存在且任务未失败，返回已有任务。
- 如果 key 已存在但参数 hash 不一致，返回 `VALIDATION_ERROR`。

### 5.2 request_hash

request_hash 由以下字段生成：

```text
taskType + providerId + modelId + prompt/messages + fileIds + params
```

用途：

- 防止重复扣费。
- 提示用户短时间内重复提交。
- 支持 rerun 时复用原始参数。

## 6. Worker 轮询规则

Worker 获取任务条件：

```text
status IN ('submitted', 'processing')
AND poll_after <= now()
AND task not locked
```

处理步骤：

1. 获取任务锁。
2. 读取 generation_task。
3. 校验当前状态仍可轮询。
4. 调用 Provider Adapter 查询状态。
5. 映射 Provider 状态。
6. 更新 PostgreSQL。
7. 如果 completed，下载或记录输出。
8. 释放锁。

锁：

```text
lock:generation_task:{taskId}
```

锁超时必须小于任务轮询间隔。

---

## 7. 轮询退避

默认策略：

| 任务年龄 | 轮询间隔 |
|---|---|
| 0-2 分钟 | 5 秒 |
| 2-10 分钟 | 15 秒 |
| 10-60 分钟 | 60 秒 |
| 60 分钟以上 | 5 分钟 |

Provider 限流时：

- 记录 `PROVIDER_RATE_LIMIT`。
- poll_after 延后。
- 不立即标记 failed。

Provider 超时时：

- 单次查询超时不等于任务失败。
- 连续超时达到阈值后标记 failed 或等待下次轮询。

## 8. 超时与过期

### 8.1 本地超时

本地任务必须有 `expires_at`。

超过 `expires_at`：

- 如果状态是 `submitted` 或 `processing`，转为 `expired`。
- 记录 error_type：`PROVIDER_TIMEOUT` 或 `TASK_EXPIRED`。

### 8.2 火山视频生成过期

火山视频生成支持 `execution_expires_after`，默认 172800 秒，取值范围来自 Provider 文档。

本系统规则：

- 如果用户未指定，使用 paramsSchema 默认值。
- 本地 `expires_at` 应不晚于 Provider 过期时间。
- Provider 返回 `expired` 时同步标记本地 `expired`。

## 9. 取消语义

取消分为两类：

```text
local_cancelled
provider_cancelled
```

如果 Provider 支持取消：

```text
POST cancel
↓
调用 Adapter cancel
↓
成功：cancelled + providerCancelled=true
失败：cancelled + providerCancelled=false + warning
```

如果 Provider 不支持取消：

```text
本地标记 cancelled
停止前端展示和本地轮询
提示 Provider 任务可能继续执行并产生费用
```

取消后的任务不能重新进入 processing。

---

## 10. 重试规则

允许自动重试：

- Provider 查询状态超时。
- Provider 临时 5xx。
- Redis 临时连接失败。
- 输出文件下载临时失败。

不允许自动重试：

- API Key 错误。
- 模型不兼容。
- 参数校验失败。
- 文件不存在。
- Provider 明确返回内容或权限错误。

重试记录必须进入 request_logs。

## 11. 文件解析状态机

状态：

```text
uploading
uploaded
parsing
parsed
failed
deleted
```

允许流转：

```text
uploading -> uploaded
uploaded -> parsing
uploaded -> deleted
parsing -> parsed
parsing -> failed
parsed -> deleted
failed -> deleted
```

只有 `parsed` 或 `direct_usable=true` 的 `uploaded` 文件可以进入任务执行。

---

## 12. 验收标准

- Chat Run 不能从 completed 回到 running。
- Generation Task 不能从 completed 回到 processing。
- Worker 更新任务前必须加锁。
- Provider 原始状态保存到 `provider_status`。
- Redis 清空不影响 PostgreSQL 中的最终状态。
- 重复 Idempotency-Key 不会创建第二个收费任务。
- Provider 不支持取消时，前端明确提示本地取消和 Provider 取消的区别。
