# 文件上传与解析设计文档

项目名称：ModelGate  
版本：v1.0  
适用范围：文件上传、文件安全、文件解析、预览、模型输入准备

---

## 1. 目标

文件系统要支持多模态任务，同时保持安全边界清晰。

目标：

- 支持图片、视频、音频、文档和代码文件。
- 文件上传不造成路径穿越、XSS 或资源耗尽。
- 文件解析结果可参与模型推荐和任务执行。
- 前端不接触服务器物理路径。
- Adapter 不处理文件安全问题。

---

## 2. 支持文件类型

| 类型 | 扩展名 | detectedType |
|---|---|---|
| 图片 | jpg, jpeg, png, webp, gif | image |
| 视频 | mp4, mov, webm | video |
| 音频 | mp3, wav, m4a | audio |
| 文档 | pdf, docx, txt, md, csv, xlsx, pptx | document |
| 代码 | py, js, ts, tsx, html, css, java, cpp, go, rs, json, yaml | code |

### 2.1 第一版实现范围

Phase 5 第一版优先实现：

| 类型 | 第一版处理 | 状态 |
|---|---|---|
| 图片 | 扩展名 / MIME / 文件头校验，metadata 提取，缩略图 / preview 生成 | 必做 |
| PDF | 文本提取，页数 metadata，按页片段 metadata | 必做 |
| DOCX | 段落与表格文本提取，标题层级尽量保留 | 必做 |
| TXT / MD | 文本读取，标题 / 代码块基础识别 | 必做 |
| 代码文件 | 语言识别、行数、文本读取，不执行代码 | 必做 |
| CSV | 表头、行列数量、前 N 行样本 | 可做 |
| PPTX / XLSX | 先做 metadata 与基础文本，解析质量后续增强 | 可做 |
| 视频 / 音频 | 第一版只保存原文件和基础 metadata；抽帧 / 转写后续增强 | 预留 |

第一版不支持：

- 可执行文件。
- 压缩包自动解压。
- Office 宏执行。
- 上传插件并执行。
- HTML / SVG inline 执行。

HTML 文件可以作为代码 / 文本分析输入，但预览接口必须按 attachment 返回，不能在浏览器 inline 渲染。

## 3. 上传流程

```text
前端选择文件
↓
POST /api/files/upload
↓
后端生成 fileId
↓
校验大小、扩展名、MIME、文件头
↓
保存到 storage/uploads
↓
写入 files: uploaded
↓
需要解析则投递 Worker
↓
Worker 解析并写回 metadata
↓
files: parsed 或 failed
```

---

## 4. 文件命名与路径

禁止：

- 使用用户原始文件名作为真实路径。
- 使用用户输入拼接路径。
- 返回服务器物理路径。

存储命名：

```text
storage/uploads/{yyyy}/{mm}/{fileId}.{safeExt}
storage/previews/{yyyy}/{mm}/{fileId}.png
storage/outputs/{yyyy}/{mm}/{outputId}.{ext}
```

数据库：

```text
original_name: 用户原始文件名
stored_path: 内部路径或对象存储 key
preview_path: 预览路径或 key
```

API 只返回：

```text
/api/files/{fileId}/preview
/api/files/{fileId}
```

## 5. 安全校验

### 5.1 必须校验

- 扩展名。
- MIME。
- 文件头 magic number。
- 单文件大小。
- 总上传大小。
- 并发上传数。
- 文件名长度。

校验顺序：

```text
读取请求头 Content-Length
↓
限制单文件大小和总上传大小
↓
提取原始文件名，仅作为 metadata
↓
校验扩展名白名单
↓
读取文件头 bytes
↓
校验 magic number 与 MIME 是否兼容
↓
生成 fileId 和安全存储名
↓
写入 storage/uploads
↓
写入 files 表
```

magic number 第一版规则：

| 类型 | 扩展名 | 文件头规则 |
|---|---|---|
| JPEG | jpg, jpeg | `FF D8 FF` |
| PNG | png | `89 50 4E 47 0D 0A 1A 0A` |
| GIF | gif | `GIF87a` 或 `GIF89a` |
| WEBP | webp | `RIFF....WEBP` |
| PDF | pdf | `%PDF-` |
| ZIP 容器 | docx, pptx, xlsx | `PK`，并继续校验 Office 内部结构 |
| MP4 / MOV | mp4, mov | `ftyp` box |
| MP3 | mp3 | `ID3` 或 MPEG frame sync |
| WAV | wav | `RIFF....WAVE` |
| 文本 / 代码 | txt, md, py, js 等 | UTF-8 / UTF-16 可解码，拒绝明显二进制内容 |

MIME 校验不能单独作为可信依据，必须与扩展名和文件头一起判断。

### 5.2 建议默认限制

| 类型 | 单文件上限 |
|---|---|
| image | 20 MB |
| video | 500 MB |
| audio | 100 MB |
| document | 100 MB |
| code | 5 MB |

这些值属于第一版建议，可在开发前决策中确认。

总上传限制：

```text
单次请求最多 1 个文件
单用户本地 storage 总量默认 5 GB
单次解析读取内存不超过文件大小上限，优先流式读取
```

### 5.3 响应头

预览与下载接口建议：

```http
X-Content-Type-Options: nosniff
Content-Disposition: inline; filename="safe-name"
```

对 HTML / SVG：

```http
Content-Disposition: attachment
```

## 6. 文件状态

```text
uploading
uploaded
parsing
parsed
failed
deleted
```

规则：

- `uploading` 表示前端选择文件后到后端响应前的 UI 临时态，不落库也可以。
- `uploaded` 表示文件保存成功。
- `parsing` 表示 Worker 正在解析。
- `parsed` 表示解析完成，可进入模型推荐。
- `failed` 表示解析失败，但可根据任务决定是否仍可使用原文件。
- `deleted` 表示逻辑删除。

状态流转：

```text
uploading(UI)
  → uploaded
  → parsing
  → parsed

uploading(UI)
  → failed

uploaded
  → deleted

parsed
  → deleted

failed
  → deleted
```

进入模型推荐条件：

```text
status = parsed
OR
status = uploaded AND direct_usable = true
```

## 7. 图片处理

提取 metadata：

- width。
- height。
- format。
- color mode。
- has alpha。
- file size。

生成：

- preview png。
- thumbnail。

第一版 preview 规则：

```text
storage/previews/{yyyy}/{mm}/{fileId}.webp
最大边 1024px
保留原图比例
去除 EXIF 中可能包含的定位信息
```

作为 Provider 输入：

- 支持 URL 的 Provider：使用 signed URL 或可访问对象存储 URL。
- 支持 base64 的 Provider：生成 data URL。
- 支持素材库的 Provider：后续可上传为 provider asset ID。

安全：

- SVG 不 inline 渲染。
- GIF 只作为图片输入，不执行脚本。

---

## 8. 视频处理

提取 metadata：

- duration。
- width。
- height。
- fps。
- codec。
- audio codec。
- container。
- file size。

生成：

- poster frame。
- thumbnails。
- 抽帧结果，带时间戳。

抽帧策略：

```text
短视频：均匀抽 5-10 帧
长视频：按场景或固定间隔抽帧
```

第一版可先做固定间隔抽帧。

Provider 输入：

- 火山 Seedance 2.0 后续接入时支持视频 URL 或素材 ID；第一版暂不接入 Seedance。
- 如果 Provider 不支持直接视频输入，则抽帧后走 vision_understanding。

---

## 9. 音频处理

提取 metadata：

- duration。
- format。
- sample rate。
- channels。
- bitrate。

第一版策略：

- 保存原文件。
- 可为后续 audio_transcription 预留。
- 如果 Provider 支持音频输入，可转换为 URL 或 base64 data URL。

火山 Seedance 2.0 后续接入时可接受音频作为部分多模态参考，但不能单独只输入音频生成视频；第一版暂不接入 Seedance。

## 10. 文档处理

### 10.1 PDF

解析：

- 文本。
- 页码。
- 页面截图，按需。
- 表格，按能力提取。

metadata：

- page_count。
- title。
- author。
- encrypted。

解析输出：

```json
{
  "parsedText": "...",
  "chunks": [
    {
      "index": 0,
      "source": {
        "fileId": "file_123",
        "page": 1
      },
      "text": "..."
    }
  ],
  "metadata": {
    "pageCount": 12
  }
}
```

### 10.2 DOCX

解析：

- 段落。
- 标题层级。
- 表格。
- 图片引用信息。

不执行：

- 宏。
- 外部链接。

### 10.3 PPTX

解析：

- slide number。
- 文本框。
- 备注。
- 图片 metadata。

### 10.4 XLSX / CSV

解析：

- sheet 名。
- 行列数量。
- 表头。
- 前 N 行样本。
- 可转换为 Markdown 表格或 JSON。

### 10.5 TXT / MD

解析：

- 纯文本。
- Markdown 标题。
- 代码块。

## 11. 代码文件处理

提取：

- language。
- line count。
- file extension。
- text content。

限制：

- 不执行代码。
- 不安装依赖。
- 不解析远程 import。

上下文组织：

```text
BEGIN_CODE_FILE path="src/app.ts" language="typescript"
...
END_CODE_FILE
```

语言识别优先级：

```text
扩展名映射
↓
文件名特例，例如 Dockerfile、Makefile
↓
shebang，例如 #!/usr/bin/env python
↓
unknown
```

代码文件进入模型前必须保留边界符，不允许直接拼入 system prompt。

---

## 12. 文件内容进入 Prompt 的边界

文件内容必须作为用户上下文：

```text
BEGIN_USER_FILE_CONTEXT fileId="file_123"
...
END_USER_FILE_CONTEXT
```

禁止：

- 拼入 system prompt。
- 让文件内容覆盖系统指令。
- 让文件内容触发本地命令。

## 13. Worker 解析任务

任务类型：

```text
parse_image
parse_video
parse_audio
parse_document
parse_code
generate_preview
```

第一版执行策略：

- 小文件图片 metadata、TXT/MD/代码文本识别可以在上传请求内同步完成。
- PDF / DOCX / PPTX / XLSX 解析走 Worker，避免阻塞 FastAPI 请求。
- 视频抽帧、音频 metadata 后续版本再做；第一版只保存文件并记录基础 metadata。
- Worker 写回 `files.metadata_json`、`files.preview_path`、`files.status`、`files.error_message`。

Worker 输出：

```json
{
  "metadata": {},
  "previewPath": "storage/previews/2026/05/file_123.png",
  "chunks": [],
  "error": null
}
```

`metadata_json` 结构建议：

```json
{
  "kind": "document",
  "extension": ".pdf",
  "parser": "pypdf",
  "parseVersion": 1,
  "parsedText": "...",
  "chunks": [],
  "previewUrl": "/api/files/file_123/preview",
  "source": {
    "originalName": "demo.pdf",
    "mimeType": "application/pdf",
    "sizeBytes": 12345
  }
}
```

大型文档不得把全部文本长期塞入前端状态；前端只展示摘要、片段数量和解析状态。

解析失败：

- files.status = failed。
- error_message 写入脱敏错误。
- 前端展示失败原因。

## 14. StorageService

接口：

```python
class StorageService:
    async def save_upload(file) -> StoredFile: ...
    async def get_file_stream(file_id: str) -> AsyncIterator[bytes]: ...
    async def get_preview_stream(file_id: str) -> AsyncIterator[bytes]: ...
    async def save_output(bytes_data: bytes, ext: str) -> StoredOutput: ...
    async def create_signed_url(storage_key: str) -> str: ...
```

本地实现：

```text
LocalStorageService
```

生产实现：

```text
ObjectStorageService
```

业务代码只依赖 StorageService，不硬编码本地路径。

## 15. 验收标准

- 上传文件名不能造成路径穿越。
- 上传文件使用系统生成 ID 和安全扩展名保存。
- 扩展名、MIME、文件头三者不一致时拒绝上传。
- SVG / HTML 不会被 inline 执行。
- 超大文件会被拒绝。
- 文件预览接口不返回物理路径。
- 文件 metadata 能参与模型推荐。
- 删除文件后，前端表单中的 fileIds 同步清理。
- 解析失败不会阻塞其他文件。
- Redis 清空不影响 files 表中已解析 metadata。
- PDF / DOCX / TXT / MD / 代码文件至少有一个成功解析测试。
- 图片上传后能生成 preview，并且 preview 不包含原始 EXIF 定位信息。
- 解析内容进入模型时必须带 `BEGIN_USER_FILE_CONTEXT` 边界。
