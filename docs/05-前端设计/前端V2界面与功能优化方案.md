# ModelGate 前端 V2 界面与功能优化方案

版本：v2.0 设计草案  
基于：当前 v1 前端实现、`前端页面设计.md`、`前端视觉参考改造计划.md`、`前端配色设计.md`  
目标：把 ModelGate 从“可运行的多模型 Playground”升级为“适合长期使用、便于排障、支持对比和复用的本地 AI 工作台”。

---

## 1. V1 现状判断

当前 v1 已经完成了核心工作流：

- 左侧 Sidebar：Overview、Playground、Models、Usage、Activity Logs、API Keys、Settings。
- 浅色低饱和主题：暖灰背景、鼠尾草绿主色、纸感卡片。
- Playground：任务 Tabs、模型选择、Prompt 输入、文件上传、参数 Popover、Run / Cancel、输出区。
- Overview：Provider 状态、模型数量、运行统计、能力入口、最近运行记录。
- Usage：请求数、tokens、成本、延迟、失败率和图表。
- History / Logs：运行记录、请求日志、筛选、详情抽屉、Rerun、Delete。
- Settings：Provider API Key 本地写入和清理。

V1 的问题不在于“没有页面”，而在于：

| 问题 | 表现 | V2 优化方向 |
|---|---|---|
| 工作台仍偏单次执行 | 用户运行一次后，下一步操作不够清晰 | 增加运行状态流、结果操作、历史复用、对比入口 |
| 模型选择信息不足 | 只看到模型名和 Provider，不容易判断为什么推荐 | 增加模型能力摘要、匹配原因、不可用原因、成本/延迟提示 |
| 结果区可读性有限 | 文本、图片、视频、状态、Archive 混在一个简单 Tabs 中 | 按输出类型设计 Preview，并增加 Request Summary |
| 日志排障入口分散 | History 与 Request Logs 的关系需要用户自己理解 | 建立从 Run 到 Request Log 的追踪链路 |
| Usage 更像统计页 | 统计有了，但未直接服务于模型选择和排障 | 增加趋势解释、成本异常、失败模型排行 |
| API Key 管理偏基础 | 只能保存或清理，缺少连通性测试和错误解释 | 增加 Test Connection、Key 来源、最近错误 |
| V1 标识仍存在 | Sidebar footer 仍显示 `v1.0` | 更新版本信息和 About 区域 |

## 2. V2 产品定位

V2 不建议重做成营销式首页，也不建议堆更多卡片。它应该强化三个核心场景：

1. 快速运行：用户进入 Playground 后，用最少步骤完成任务、上传文件、选择模型、运行。
2. 快速判断：用户能看懂为什么某个模型可用、不可用、便宜、慢或容易失败。
3. 快速复用：用户能从历史记录、模板、参数组合和对比结果中继续工作。

一句话定位：

```text
ModelGate V2 = Task-first Playground + Model Decision Layer + Run Diagnostics + Reusable Workspace
```

## 3. 视觉方向优化

V2 继续沿用现有“浅色、低饱和、纸感、温和科技感”的方向，不切回深色控制台。

### 3.1 保留

- 暖灰米白背景。
- 鼠尾草绿主按钮。
- 柔和边框和轻阴影。
- 8px 左右的常规组件圆角。
- 图标按钮优先，文字按钮只用于明确命令。
- Sidebar + Topbar + Main 的桌面工作台布局。

### 3.2 调整

| 区域 | V1 问题 | V2 调整 |
|---|---|---|
| 页面背景 | 点阵背景有识别度，但长时间使用可能分散注意力 | 只在 Playground 中心区域保留弱点阵，数据页面改纯背景 |
| 主卡片 | Playground 卡片较完整，但结果区层级偏弱 | 输入卡片和输出卡片保持同宽，建立“上输入、下结果”的稳定节奏 |
| 状态色 | Provider、Run、Usage 状态颜色有时不统一 | 建立统一状态规范：Ready、No Key、Running、Queued、Failed、Limited |
| 表格 | 当前列表卡片多，扫描效率一般 | Logs、Usage 明细、Models 使用更紧凑的表格 |
| 文案 | 部分页面仍是通用英文描述 | 改为更像开发者工具的短标签和字段名 |

### 3.3 推荐视觉标准

- 页面主背景：`hsl(var(--background))`，不额外加强渐变。
- 数据页容器：无外层大卡片，直接使用标题、过滤器、表格、图表区域。
- 操作区卡片：只用于 Playground 输入框、输出预览、设置表单。
- 表格行高：`40px-48px`，字段紧凑，ID 使用 monospace。
- 关键按钮：
  - Run：主色。
  - Cancel / Delete：destructive。
  - Copy / Download / Rerun：outline 或 ghost + 图标。

## 4. 信息架构 V2

V2 导航建议保持 7 个主入口，但把页面职责重新定义清楚。

| 导航 | V2 名称 | 页面目标 |
|---|---|---|
| `/` | Overview | 状态总览、近期异常、快捷进入 Playground |
| `/workspace` | Playground | 单模型运行、文件输入、参数调整、结果预览 |
| `/models` | Models | 模型注册表、能力、可用性、推荐原因调试 |
| `/usage` | Usage | 成本、tokens、延迟、失败率、趋势分析 |
| `/logs` 或 `/history` | Activity | Run、Generation Task、Request Log 的统一追踪 |
| `/api-keys` | API Keys | Provider Key 配置、连通性测试、最近认证错误 |
| `/settings` | Settings | 应用偏好、本地数据、运行环境、后续账号与登录设置 |

V2 可以保留 `/history` 兼容路由，但导航名称统一为 `Activity`，避免用户不知道 History 和 Logs 的区别。

API Keys 不应继续挂在 Settings Tab 里。Provider Key 是高频、强风险、强排障属性的独立工作页面；Settings 则应该是低频的应用配置中心。这样后续版本加入用户登录、Profile、Session、团队空间时，不需要再次拆导航结构。

## 5. Playground V2

Playground 是 V2 的第一优先级。

### 5.1 页面结构

```text
┌──────────────────────────────────────────────────────────────┐
│ Mode Tabs                                                     │
│ Chat / Coding / Review / Document / Prompt / Generation       │
│                                                              │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │ Model row: Model, Provider, Match Score, Key Status       │ │
│ │ Prompt / file input                                      │ │
│ │ File chips + parsed state                                │ │
│ │ [Upload] [Parameters] [Templates]        [Run / Cancel]  │ │
│ └──────────────────────────────────────────────────────────┘ │
│                                                              │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │ Output | Timeline | Request | Archive | Compare           │ │
│ └──────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

### 5.2 模式 Tabs

现有 Tabs 可以保留，但 V2 要增强两点：

- 每个 Tab 显示可用模型数量，例如 `Chat 4`、`Document 2`。
- 不可用任务不要只禁用，应显示原因：`No configured provider`、`No model supports video`。

推荐任务分组：

| 分组 | Tabs |
|---|---|
| Core | Chat、Coding、Code Review |
| Analysis | Document、Image、Video |
| Creative | Prompt Optimize、Image Gen、Video Gen |

桌面端可以使用两行紧凑 segmented controls；移动端使用横向滚动 Tabs。

### 5.3 模型选择行

V1 当前只显示 `Model + Provider Badge`。V2 建议改成“模型决策行”：

| 字段 | 说明 |
|---|---|
| Model Select | 官方模型名和 displayName |
| Provider | Provider 名称和 Ready / No Key 状态 |
| Match | 当前任务匹配度，例如 `Best match`、`Supports files` |
| Cost / Speed | 如果已有 usage 数据，显示 `Low cost`、`Fast recent latency` |
| Details | 点击打开模型详情 Sheet |

模型选择下拉不只列模型名，应显示：

- Provider。
- Runtime：Chat / Generation。
- 支持输入：text、file、image、video。
- 最近平均延迟。
- 最近失败率。
- 不可用原因。

### 5.4 Prompt 输入区

V2 输入区继续保持大文本框，但需要更接近真实工作流：

- 支持 `System instruction` 折叠区。
- Prompt 主输入保留 4000 字符提示，但字符统计不要抢视觉。
- 文件 chips 增加解析状态：
  - uploading
  - uploaded
  - parsing
  - parsed
  - failed
- 文档分析任务中，文件 chip 可展开查看 parsed preview。
- 运行前如果文件未解析完成，Run 按钮应显示等待或明确阻止原因。

### 5.5 参数 Popover

现有 Params Popover 方向正确，V2 主要补齐参数体验：

| 优化 | 说明 |
|---|---|
| 参数分组 | Sampling、Limits、Output、Provider、Advanced |
| 变更提示 | 修改过的字段显示小点或 `modified` 状态 |
| Preset | `Creative`、`Precise`、`Long context`、`Low cost` |
| Reset 范围 | 支持 Reset current group 和 Reset all |
| Schema 透明度 | 保留 Provider、Model、Schema version |
| 错误校验 | 参数超出范围时在 Popover 内提示，不等后端报错 |

### 5.6 模板功能

V1 已有 Template 按钮。V2 可以把模板升级成可复用资产：

- 每个任务内置 3-5 个模板。
- 用户可以保存当前 Prompt 为本地模板。
- 模板字段：
  - name
  - taskType
  - prompt
  - recommendedParams
  - fileRequired
- 选择模板时可选择：
  - 只填充 Prompt。
  - 同时应用推荐参数。

### 5.7 运行中状态

V2 应把运行状态从简单按钮状态升级为可读 timeline：

```text
Queued -> Validating input -> Calling provider -> Receiving output -> Saving result -> Completed
```

失败时显示：

- 用户可读错误：`API key invalid`、`Provider rate limited`、`File parsing failed`。
- requestId。
- Provider。
- Model。
- 可操作按钮：`Open log`、`Retry`、`Change model`、`Edit API key`。

### 5.8 Output 区

V2 输出区建议从 `Output / Status / Archive` 扩展为：

| Tab | 内容 |
|---|---|
| Output | 文本、Markdown、代码、图片、视频预览 |
| Timeline | 运行步骤、耗时、状态 |
| Request | Provider、model、params、tokens、latency、cost、requestId |
| Archive | 最近结果 |
| Compare | 对比结果入口 |

不同输出类型的展示方式：

| 输出类型 | V2 展示 |
|---|---|
| text | Markdown renderer + raw text toggle |
| code | 代码块、复制、下载 |
| image | 大图预览、下载、打开原图、重新生成 |
| video | 播放器、封面、下载、任务状态 |
| file | 文件名、大小、下载、来源 run |

## 6. Overview V2

Overview 不需要变成首页宣传页，而是工作台健康页。

### 6.1 顶部摘要

建议四个核心指标：

- Configured Providers。
- Enabled Models。
- Runs Today。
- Failure Rate。

### 6.2 健康状态区

新增 `Health Checklist`：

| 检查项 | 状态 |
|---|---|
| Provider keys configured | Ready / Missing |
| Database connected | Ready / Error |
| Redis worker connected | Ready / Error |
| Generation runtime available | Ready / Pending |
| Recent provider auth errors | Count |

### 6.3 最近异常

Overview 应把“最近失败”提前展示：

- 最近 5 个 failed runs。
- 最近 5 个 auth/rate-limit/provider errors。
- 每条提供 `Open log` 和 `Retry`。

### 6.4 快捷入口

Capability cards 保留，但改成更强的动作入口：

- Chat: `Open`
- Coding: `Open`
- Document Analysis: `Upload document`
- Prompt Optimize: `Start from template`
- Generation: 如果暂未完整支持，显示 `Preview` 或 `Coming later`，不要做成可点击但无效的卡片。

| Provider | Provider |

### 7.2 模型详情 Sheet

- paramsSchema。

- 最近 usage。
- 最近错误。

### 7.3 推荐逻辑调试

用户选择：

- taskType。
- inputTypes。
- outputType。

页面输出：

- 推荐模型列表。
- 隐藏模型列表。
- 每个模型的隐藏原因。

这可以直接复用当前 `RecommendResult` 中 `availableModels` 和 `hiddenModels` 的概念。

## 8. Usage V2

Usage 页面已有基础图表，V2 要让它更能辅助决策。

### 8.1 指标升级

保留：

- Total requests。
- Total tokens。
- Estimated cost。
- Average latency。
- Failure rate。

新增：

- Most used model。
- Most failed model。
- Slowest provider。
- Cost per successful run。

### 8.2 图表优化

| 图表 | V2 用途 |
|---|---|
| Requests by day | 看使用量趋势 |
| Tokens by provider | 看资源消耗分布 |
| Latency trend | 看 Provider 是否变慢 |
| Failures by error type | 看问题主要来源 |
| Cost by model | 判断高成本模型 |

### 8.3 异常解释

Usage 页面增加小型 insight 区：

- `MiniMax failed 6 times in the last 7 days due to quota or rate limit.`
- `MiMo has the lowest recent latency for chat tasks.`
- `Document analysis consumes most tokens this week.`

第一版可以用简单规则生成，不需要复杂 AI 分析。

## 10. API Keys V2

当前 Settings 已能保存 API Key，但 V2 需要把 API Keys 从 Settings 中拆出来，形成独立页面 `/api-keys`。原因是 API Key 管理属于 Provider 连接能力，不是普通偏好设置；它和 Playground、Models、Activity 的错误排障直接相关。

### 10.1 页面定位

API Keys 页面只处理一件事：让用户知道每个 Provider 是否已经配置、是否可连通、最近为什么失败。

不放在 API Keys 页面的内容：

- 主题切换。
- 本地缓存清理。
- 登录账号资料。
- 数据导入导出。
- 通用应用设置。

### 10.2 Provider Key 卡片

每个 Provider 显示：

- Provider name。
- Enabled / Disabled。
- Configured: Env / Local UI / Missing。
- Masked key fingerprint，例如 `sk-...a91c`，不显示明文。
- Last tested at。
- Last error。

建议卡片布局：

```text
┌──────────────────────────────────────────────────────────────┐
│ MiMo                                      Ready / Local Key   │
│ Base URL: https://...                    Last test: 2m ago   │
│ Key: sk-...a91c                          Last error: none    │
│ [Test] [Update key] [Clear local key]                        │
└──────────────────────────────────────────────────────────────┘
```

### 10.3 Test Connection

新增按钮：

- `Test`
- `Save`
- `Clear`

Test 逻辑：

- 调用轻量 health/model endpoint。
- 如果 Provider 不支持 health，就执行最小安全请求。
- 显示结果：
  - Connected。
  - Invalid key。
  - Quota exhausted。
  - Rate limited。
  - Endpoint unreachable。

### 10.4 从其他页面跳转

V2 中 API Keys 应成为错误排障链路的一部分：

| 来源页面 | 跳转触发 |
|---|---|
| Playground | Run 失败，错误为 invalid_api_key、missing_api_key |
| Models | 某 Provider 下所有模型不可用 |
| Activity | Request log 显示 auth failed |
| Overview | Health Checklist 显示 Provider key missing |

跳转到 `/api-keys?provider=mimo` 时，应自动定位对应 Provider 卡片。

### 10.5 安全提示

保留短提示即可：

```text
Keys are stored locally. Plaintext is never shown after saving.
```

不要在页面上写过长说明。

## 11. Settings V2

Settings 页面不再放 Provider API Key。它的职责是管理应用级设置、本地数据、运行环境，以及为后续登录版本预留账号设置入口。

### 11.1 Settings 页面结构

Settings 建议采用左侧二级菜单或顶部 Tabs。桌面端推荐左侧二级菜单，因为后续登录后设置项会增加。

| Tab | 内容 |
|---|---|
| General | 默认启动页、默认任务类型、语言、时间格式 |
| Appearance | Light / System，后续可加 Dark，密度设置 |
| Workspace | 默认模型、默认参数、Prompt 模板位置、下载目录 |
| Storage | uploads、outputs、previews、history 的本地清理策略 |
| Runtime | server、worker、database、redis 状态，只读诊断 |
| Privacy & Security | 本地数据保留、日志脱敏、导出前清理敏感字段 |
| Account | 后续登录版本的 Profile、Session、Logout、团队空间入口 |
| About | version、local mode、docs links、license |

### 11.2 General

General 放低风险、低频但会影响整体体验的设置：

- Default landing page：Overview / Playground。
- Default task type：Chat / Coding / Document Analysis。
- Language：English / 中文，第一版可以只显示不可编辑。
- Time display：local time / ISO。
- Confirm before destructive actions：删除历史、清理缓存前是否二次确认。

### 11.3 Appearance

Appearance 只管视觉偏好：

- Theme：Light / System，Dark 后续预留。
- Interface density：Comfortable / Compact。
- Sidebar default：Expanded / Collapsed。
- Reduce background effects：关闭 Playground 点阵和弱渐变。

### 11.4 Workspace

Workspace 处理“新任务默认怎么打开”：

- Default provider。
- Default model per task type。
- Default parameter preset。
- Prompt template storage。
- Auto-save draft prompt。
- After run behavior：stay on output / open timeline / open request summary。

这些不应该放到 API Keys 页面，因为它们是使用偏好，不是连接配置。

### 11.5 Storage

Storage 管理本地文件和历史数据：

- Uploads size。
- Outputs size。
- Previews size。
- History records count。
- 清理按钮：
  - Clear failed uploads。
  - Clear generated previews。
  - Clear old outputs。
  - Clear all local history。

危险操作必须二次确认，并明确不会清理 API Keys，避免用户误删连接配置。

### 11.6 Runtime

Runtime 是只读诊断页，不做复杂配置：

- Server status。
- Database status。
- Redis status。
- Worker status。
- Storage path status。
- Current API base URL。
- Build/version info。

这里可以提供 `Copy diagnostics`，方便用户提 issue。

### 11.7 Privacy & Security

该页面为本地单用户版本和后续登录版本都需要：

- Log request body：on / off，默认 off 或 masked。
- Mask API keys in diagnostics：固定 on。
- Include prompt in exported logs：off by default。
- Local data retention：never auto-delete / 30 days / 90 days。
- Export data。
- Delete local data。

注意：Provider API Key 的新增、更新、测试仍然在 `/api-keys`，这里最多放安全策略，不放具体 Key 输入框。

### 11.8 Account 预留

## 16. Compare（多模型对比）详细设计

Compare 是 V2 P2 / Phase V2.4 的核心，本节把 5.8 Output Tabs 的 `Compare` 入口落到可实施颗粒度。

### 16.1 触发入口

Compare 有 3 个入口，最终都打开同一个 `CompareDrawer`：

| 入口 | 触发 | 预填 |
|---|---|---|
| Playground Output → Compare Tab → `Open Compare` | 当前 Run 完成后 | 当前 Prompt / Params / 主模型 |
| Playground → 模型选择行右侧 `Compare` 图标按钮 | 任意时刻 | 当前 Prompt / Params / 主模型 |
| Activity 详情抽屉 → `Compare with another model` | 任意 run | 该 run 的 Prompt / Params / 模型 |

### 16.2 CompareDrawer 布局

```text
┌──────────────────────────────────────────────────────────────┐
│ Compare run                                          [X]      │
├──────────────────────────────────────────────────────────────┤
│ Prompt   (shared, readonly preview, edit ↗ jumps to Playground)│
│ ┌──────────────────────────────────────────────────────────┐ │
│ │ multiline prompt ...                                     │ │
│ └──────────────────────────────────────────────────────────┘ │
│ Params   [Shared] [Per-model]                                 │
│ Models   [+] up to 3                                          │
│ ┌────────────────┬────────────────┬────────────────┐          │
│ │ Slot A         │ Slot B         │ Slot C         │          │
│ │ mimo-pro    [x]│ minimax-abab[x]│ + Add model    │          │
│ │ Ready          │ Ready          │                │          │
│ │ params: default│ params: t=0.2  │                │          │
│ └────────────────┴────────────────┴────────────────┘          │
│                                          [Cancel] [Run all]  │
├──────────────────────────────────────────────────────────────┤
│ Results (after Run all):                                      │
│ ┌────────────────┬────────────────┬────────────────┐          │
│ │ Slot A         │ Slot B         │ Slot C         │          │
│ │ Status: ok     │ Status: failed │ Status: ok     │          │
│ │ 1.2s · 312 tok │ —              │ 0.9s · 198 tok │          │
│ │ $0.0021        │ —              │ $0.0014        │          │
│ │                │                │                │          │
│ │ Output text... │ Provider rate  │ Output text... │          │
│ │                │ limited        │                │          │
│ │ [Copy] [Rerun] │ [Open log]     │ [Copy] [Rerun] │          │
│ └────────────────┴────────────────┴────────────────┘          │
│ Metrics row (sortable: Latency / Tokens / Cost)               │
└──────────────────────────────────────────────────────────────┘
```

### 16.3 数量与约束

- **最多 3 个模型同时对比**（用户选择：并行调用）。理由：3 列在 1280px 桌面屏宽下每列约 380px，可读；4 列会触发横向滚动。
- 至少 2 个模型才能 Run。
- Compare 仅支持 `chat` / `chat_completion` runtime 的 Task（chat、coding、code review、document、prompt optimize）；不支持 generation（图像/视频）任务，因为输出长度和成本差异过大，并排对比意义有限。

### 16.4 运行模式（并行）

- 前端在点 `Run all` 时用 `Promise.all` 并行向后端 `POST /api/runs` 发 N 个独立请求，每个请求带：
  ```json
  {
    "task_type": "chat",
    "model_id": "...",
    "prompt": "shared",
    "params": { ... },
    "compare_group_id": "uuid-v4"
  }
  ```
- `compare_group_id` 是前端生成的 UUID，让后端在 `runs.metadata` 里打 tag，方便 Activity 页之后过滤"同一个 Compare 产生的 runs"。
- 后端**不**新增 endpoint；现有 `POST /api/runs` 接受 `compare_group_id` 作为可选字段写入 `runs.metadata.compare_group_id`（已有 jsonb 字段，零 schema 变更）。
- 单个 slot 失败不影响其他 slot；每个 slot 独立显示状态（`pending` → `running` → `ok` / `failed`）。
- 取消：Drawer 右上有 `Cancel all`；点击后调用后端 `DELETE /api/runs/{id}` 批量，已有 endpoint。

### 16.5 Params 模式：Shared vs Per-model

- **Shared（默认）**：所有 slot 共享一份 Params；改一个所有 slot 同步。
- **Per-model**：每个 slot 独立 Params；点击 slot 卡上的 `Adjust params` 打开 Popover 仅作用于该 slot。

切换 Per-model 时，当前 Shared params 作为各 slot 初始值。再切回 Shared 时给一次确认（"Reset to first slot's params?"），避免误丢。

### 16.6 结果区设计

每个 slot 卡片自上而下：

1. **Header**：Provider Badge + Model 名 + StatusPill。
2. **Metrics**：Latency / Tokens (in/out) / Cost / requestId。
3. **Output**：复用 Playground `OutputViewer`（Markdown / Code / Image / Video 按 outputType 切换），但默认折叠到 6 行 + `Expand`。
4. **Actions**：`Copy`、`Rerun this slot`（重发该 slot）、`Open in Playground`（把该 slot 作为主模型回到 Playground）、`Open log`（跳 `/activity?runId={id}`）。

### 16.7 Metrics 排序行

3 个 slot 下方有一行汇总：

| 列 | 说明 |
|---|---|
| Latency | 升序排序按钮，最快的高亮 |
| Tokens (total) | 升序排序按钮 |
| Cost | 升序排序按钮，最低的高亮 |
| Status | failed slot 单独 callout |

排序仅影响视觉顺序（用 CSS `order`），不影响 slot index（A/B/C）保持稳定，方便后续 Rerun。

### 16.8 持久化

- Compare run 本身已作为独立 runs 持久化（带 `compare_group_id` tag）。
- Compare 配置（model slots + params 模式）**不**持久化；关闭 Drawer 即丢。
- 想"复现"一次 Compare：从 Activity 页过滤 `compareGroupId=xxx`，按 timestamp 拿出 N 个 run；下个 task 加 `Reopen as compare` 按钮。本次实施只到 tag 写入。

### 16.9 与现有组件的复用

- `ModelSelectorRow` 改造为可复用组件（已是），slot 内嵌入轻量版（仅 Model + Provider Badge，无 Match/Cost/Speed 列以节省宽度）。
- `OutputViewer` / `RunTimeline` / `RequestSummary` / `ErrorBanner` 全部复用。
- `StatusPill` 复用。

### 16.10 实施切片

| 切片 | 内容 |
|---|---|
| C1 | `CompareDrawer` 组件 + 模型 slot 选择 + Shared params 模式 |
| C2 | `Run all` 并行触发 + slot 状态机 + 结果展示 |
| C3 | Per-model params + Metrics 排序行 + Rerun slot |
| C4 | `compare_group_id` 后端写入（runs.metadata） + Activity 过滤 |

本次任务实施到 C1+C2（MVP，覆盖核心场景）；C3/C4 可作为下次 task。

## 17. Prompt 模板系统详细设计

把 5.6 节的"模板字段"扩成完整可实施设计。

### 17.1 存储位置：localStorage

用户决策：本地浏览器 localStorage，键 `modelgate:prompt-templates`。

- 优点：零后端改动、零跨设备同步复杂度、与现有 `modelgate:provider-test-results` 同构。
- 限制：换设备 / 清缓存丢失。给用户 `Export` / `Import` JSON 文件补救。
- 未来登录版本上线后，再独立任务把 localStorage 迁后端（D-2026-06-04-13 风格的"暂不动"策略）。

### 17.2 数据结构

```typescript
interface PromptTemplate {
  id: string;              // uuid-v4
  name: string;            // 用户起的名字，max 60
  taskType: TaskType;      // chat / coding / code_review / document / prompt_optimize
  prompt: string;          // 主 Prompt 内容
  systemInstruction?: string; // 可选 system instruction
  recommendedParams?: Record<string, unknown>; // 可选，覆盖默认 params
  fileRequired: boolean;   // 是否要求挂载文件（document task）
  builtin: boolean;        // true 表示内置，false 表示用户保存
  createdAt: string;       // ISO timestamp
  updatedAt: string;
}

interface TemplateStore {
  version: 1;
  templates: PromptTemplate[];
}
```

### 17.3 内置模板（builtin: true）

每个任务类型预置 3-5 个，第一版：

| TaskType | 模板名 | 用途 |
|---|---|---|
| chat | General Q&A | 通用问答 |
| chat | Brainstorm ideas | 头脑风暴 |
| chat | Summarize text | 概括文本 |
| coding | Implement function | 实现单函数 |
| coding | Debug error | 报错排查 |
| coding | Refactor for readability | 可读性重构 |
| code_review | PR review | 代码审查 |
| code_review | Security audit | 安全审查 |
| document | Extract key facts | 提取要点 |
| document | Compare two documents | 文档对比 |
| prompt_optimize | Make prompt more specific | 增强具体度 |
| prompt_optimize | Add structured output | 加结构化输出 |

内置模板**不可编辑、不可删除**，但用户可以 `Save as` 复制成 user 模板再改。

### 17.4 UI 入口

#### Playground 输入区

替换现有 `Templates` 按钮为下拉：

```text
┌─────────────────────────────────────┐
│ Templates ▼                          │
├─────────────────────────────────────┤
│ Builtin                              │
│   ▸ General Q&A                      │
│   ▸ Summarize text                   │
│   ...                                │
│ Saved                                │

```

#### Manage templates 模态

| 字段 | 操作 |
|---|---|
| name | 编辑 |
| taskType | 只读（创建时决定，不允许跨 task type 迁移） |
| prompt | 编辑 |
| systemInstruction | 编辑 |

底部：

- `Import JSON`：上传 `.json`，按 `TemplateStore` schema 解析；冲突时 prompt 用户 `Skip` / `Overwrite` / `Keep both (rename)`.
- `Export all`：下载 `modelgate-templates-{ISO}.json`，只导出 `builtin: false` 的用户模板。

### 17.5 保存当前为模板

Playground 当前 Prompt + system instruction + 当前 params + taskType 一次性收集，弹模态：

```text

  TaskType: chat (auto-detected, read-only)
```

保存后写入 localStorage，下拉立即刷新。

### 17.6 名称冲突 / 校验

- 用户模板 name 在同一 taskType 内**不允许重复**；冲突时给 `Name already exists. Choose another or overwrite.`
- name 长度限制 1-60 字符；超出截断 + 提示。
- prompt 长度限制 1-32000 字符；超出阻止保存。
- recommendedParams 必须能通过当前 model schema 校验（用 ajv），否则提示具体字段错。

### 17.7 错误恢复

localStorage 解析失败时（手动改坏 JSON / 跨版本 schema 变化）：

1. 立刻 `console.warn`。
2. UI 显示 toast：`Templates corrupted. Backup saved as modelgate:prompt-templates.bak. Restored builtin only.`
3. 把损坏的字符串移到 `modelgate:prompt-templates.bak`，重置主键为只含 builtin。

## 18. 参数 Preset 详细设计

5.5 节列了 4 个 Preset 名字，本节给出每个 Preset 的字段映射、适用模型范围、UI 行为。

### 18.1 4 个内置 Preset

| Preset | 适用任务 | 设计意图 | 推荐字段值 |
|---|---|---|---|
| **Creative** | chat、prompt_optimize | 高发散，鼓励多样性 | `temperature: 1.0`、`top_p: 0.95`、`presence_penalty: 0.6`、`frequency_penalty: 0.3` |
| **Precise** | coding、code_review、document | 低发散，确定性输出 | `temperature: 0.2`、`top_p: 0.8`、`presence_penalty: 0`、`frequency_penalty: 0` |
| **Long context** | document、chat（长对话） | 适合长输入/长输出场景 | `temperature: 0.5`、`top_p: 0.9`、`max_tokens: <provider 上限的 80%>`、不动 penalty |
| **Low cost** | 任意 | 控制输出长度以省 token | `temperature: 0.5`、`top_p: 0.9`、`max_tokens: 256` |

字段值是 **target**，应用时按下述策略落到具体模型。

### 18.2 字段映射策略

不同 Provider 的 params schema 可能不一致（OpenAI 有 `presence_penalty`，Anthropic 没有；MiMo 的 `max_tokens` 上限是 8192，MiniMax 是 65536）。Apply Preset 时：

1. 拿当前模型的 `paramsSchema`（已有，`/models/{id}` 返回）。
2. 对 Preset target 的每个字段：
   - 字段存在于 schema → 设值；如超出 schema 的 `minimum`/`maximum`，clip 到边界。
   - 字段不存在于 schema → skip，不报错。
3. `max_tokens` 的"上限的 80%" 用 `schema.properties.max_tokens.maximum * 0.8`，向下取整到 256 的倍数。

应用结果在 Params Popover 上方显示一行 `Applied "Precise" preset · 4 fields set, 1 skipped (presence_penalty not supported)`。

### 18.3 UI：Params Popover 顶部 Preset 行

```text
│          │ Creative                        │      │
```

## 20. Rerun 模型选择器（Activity 详情抽屉）

9.3 节 `Rerun with another model` 按钮的具体行为。

### 20.1 触发与跳转

- 入口：Activity 详情抽屉底部 `Rerun with another model` 按钮。
- 点击后：**不**在抽屉里弹独立选择器；直接跳转 `/workspace?fromRun={runId}`，参数：
  - `fromRun=`：原 run id。
  - 跳转后 Playground 自动：
    1. 切换 taskType 到原 run 的 taskType（如 chat → chat）。
    2. 把 prompt / system instruction / params 预填到输入区。
    3. 文件附件**不**自动重传（用户需要重新上传）；显示 toast "Original files not re-attached. Upload again if needed."
    4. ModelSelectorRow 进入"高亮选择"状态：原 model 标 `Original`，其他可选模型按现有 Match / Cost / Speed 排序，用户主动改选。
- 选择新模型后，Run 流程与普通 Playground 完全一致。

### 20.2 为什么不在抽屉里做选择器

- ModelSelectorRow 是带 Match / Cost / Speed / Details Sheet 的复合组件，复用到抽屉里会膨胀抽屉宽度。
- 切到 Playground 后用户能完整看到 Prompt / Files / Params 并继续编辑，比抽屉内"半选好" UX 更连贯。
- "原 model 标 `Original`" 是轻量做法，无需重复实现选择器。

### 20.3 `Rerun same model` 区别

- 同抽屉的另一按钮 `Rerun same model`：直接调 `POST /api/runs` 用原 run 全部字段，**不**离开抽屉；新 run 完成后 toast `Rerun completed`，点击 toast 跳到新 run 的详情。
- 适用于"我只是想重试一次看是否偶发错误"场景。

## 21. Privacy & Security 的 Export / Delete 格式

11.7 节 `Export data` / `Delete local data` 的字段细节。

### 21.1 Export

按钮触发后弹模态：

```text
Export local data
  Scope:
    ☑ Run history (runs + outputs)
    ☑ Request logs
    ☑ Usage logs
    ☐ Prompt templates (user-saved only)
    ☐ Provider configuration (excludes API keys)
  Format:  ( ) JSON (single file)
           ( ) ZIP (one .json per scope + manifest.json)
  Date range:
    [ All time ]  [ Last 30 days ]  [ Last 90 days ]  [ Custom... ]
  ☑ Mask prompt / output text (replace with length only)
  ☑ Mask file content (replace with filename + size)
  Note: API keys are NEVER exported.
[Cancel] [Export]
```

格式细节：

- **JSON 模式**：单文件 `modelgate-export-{ISO}.json`，顶层：
  ```json
  {
    "schemaVersion": 1,
    "exportedAt": "2026-06-07T12:34:56Z",
    "modelgateVersion": "v2.x",
    "scopes": {
      "runs": [...],
      "requestLogs": [...],
      "usageLogs": [...],
      "templates": [...]
    },
    "manifest": { "runs": 142, "requestLogs": 387, ... }
  }
  ```
- **ZIP 模式**：`modelgate-export-{ISO}.zip` 包含：
  - `manifest.json`（schemaVersion / exportedAt / counts）
  - `runs.json` / `request_logs.json` / `usage_logs.json` / `templates.json`（视 scope 勾选）
  - `README.txt`：解释格式 + masking 状态。
- `runs.json` 等字段对齐当前 `/api/usage/logs/{id}` 的 `_serialize_usage_log` 结构，保持 contract 一致。
- Masking：勾选时把 `prompt`、`output`、`file.content` 替换为 `"[masked: 1234 chars]"` / `"[masked: 2.3MB file.pdf]"`，方便分享给排障对象时不泄露内容。

### 21.2 Delete local data

按钮触发后弹模态：

```text
Delete local data
  This will permanently remove:
    ☑ Run history (142 runs)
    ☑ Request logs (387 logs)
    ☑ Usage logs (142 aggregates)
    ☑ Generated previews (24 files, 18.3 MB)
    ☐ Prompt templates (5 user-saved)
    ☐ Provider configuration (12 keys)
  Date range:
    ( ) All time
    ( ) Older than 30 days
    ( ) Older than 90 days
  Type "DELETE" to confirm: [__________]
[Cancel] [Delete]
```

- 不勾选默认包含的 4 项，要求用户主动勾选；与 Export 不同，Delete 是高风险二次确认。
- 必须输入 `DELETE` 全大写才能点确认按钮（防误点）。
- 删除走后端批量 endpoint（沿用 Activity 现有 `DELETE /api/usage/logs/{id}`，但加 `DELETE /api/usage/logs?olderThan=...`）；这是少数本任务**确实**需要后端改动的点。
- 删除完成后 toast `Deleted N records and M MB. API keys preserved.`，强调 API keys 没动。

### 21.3 隐私默认

- Export 默认勾 `Mask prompt / output text` + `Mask file content`。用户主动取消才会导出明文。
- 任何 Export 都**绝对不**包含：
  - API key 明文（即使用户开了 `Include provider configuration`，导出的是 `{providerId, baseUrl, enabled}`，不含 `apiKey`）。
  - `MODELGATE_SECRET_KEY` / 加密密钥相关字段。
  - 本地用户系统路径（已被 `app.core.logging.redact` 脱敏的字段保持脱敏态）。

## 22. Test Connection — 各 Provider 探测策略

10.3 节 `Test Connection` 按钮的具体实现已落在 `POST /api/providers/{id}/test`（D-2026-06-06-01）。本节补 3 类 Provider 各自的最小探测细节，新增 Provider 时按表照抄。

### 22.1 探测路径选择

| Adapter 类型 | 探测路径 | 调用形式 |
|---|---|---|
| OpenAI-compatible（mimo、deepseek、moonshot 等） | `POST {base_url}/chat/completions` | `chat()` 复用，挑 first enabled `runtime ∈ {chat, chat_completion}` 模型，`messages=[{role:user, content:"ping"}]`、`max_tokens=1`、`stream=false` |
| Anthropic-compatible（minimax-anthropic 等） | `POST {base_url}/v1/messages` | `chat()` 复用，同上 + 自动带 `anthropic-version: 2023-06-01` header |
| Volcengine Seedance（视频生成，无 chat） | `GET {base_url}/contents/generations/tasks/__modelgate_probe__` | 一个肯定不存在的 task id；用 GET 而非 POST，避免实际创建任务消耗配额 |

### 22.2 错误分类规则（沿用 `_classify_probe_error`）

3 类 Provider 都通过同一个 `_classify_probe_error` 映射。Volcengine 的 GET 不存在 task 时返回 4xx：

- 401 → `auth_failed`（鉴权失败 = key 错）
- 403 → `forbidden`
- 404 → `ok`（鉴权通过、endpoint 可达；404 本身就是探测预期，因为我们用了假 task id）
- 429 → `rate_limited`
- 5xx → `server_error`
- 网络/超时 → `unreachable` / `timeout`

注意：Volcengine 的"404 = ok"逻辑在 `test_provider_connection` 里单独走一个分支：当 `runtime ∉ chat` 且 provider 是 `volcengine_seedance` 时，跳到 GET probe 路径，把 `PROVIDER_BAD_REQUEST`(404) 重新映射为 `ok`。

### 22.3 配额与成本

- OpenAI / Anthropic 探测：1 token，成本 < $0.001。
- Volcengine 探测：GET 请求**不**计入生成配额（只有 POST create task 才计费）。

整体：1 次 Test 成本忽略不计；UI 可以"用户随时点 Test 不限频"。后端不需要加 rate limit。

### 22.4 新增 Provider 接入清单

新 Provider adapter 接入时，作者需在 PR 描述里说明：

1. 适用哪一类 probe（OpenAI / Anthropic / 自定义）。
2. 如果是自定义：probe HTTP method + path + 期望 status code mapping。
3. 在 `_pick_probe_model` 是否需要加新的 runtime 白名单。
4. 配套 pytest（参考 `test_provider_test_phase11.py` 9 条用例的覆盖度）。
